# PDF Pass/Ticket Generation

## Overview

This flow covers generating PDF passes (tickets) for token-pass holders. A pass PDF contains the token name, edition number, optional variant attributes (e.g., tier, seat, color), and a list of benefits -- each rendered with a QR code for redemption at physical or digital check-in points. The service supports both a built-in default template and fully custom HTML templates via Handlebars.

> **Internal service:** The PDF service does not require authentication. It is an internal service called by the Pass backend to generate PDFs for token-pass holders. Developers building custom applications can also call this endpoint directly from their backend services.

## Prerequisites

| Requirement | Description | How to obtain |
|-------------|-------------|---------------|
| Token pass data | Token name, edition number, and variant attributes | From the Pass & Benefits module or your application logic |
| Benefits data | Array of benefits with names, descriptions, rules, dates, and QR code data | From the Pass & Benefits module or your application logic |
| QR code data | Redemption URL or data string for each benefit | Generated by the Pass backend or your application |

## Flow: Generate Pass PDF

### Step 1: Prepare the Request Data

Build the `PassDto` with all required fields. At minimum, you need `tokenName`, `tokenEditionNumber`, an array of `tokenVariants` (can be empty), and an array of `benefits` with at least one entry.

**Minimal Request:**

```json
{
  "tokenName": "Summer Festival Pass",
  "tokenEditionNumber": "1",
  "tokenVariants": [],
  "benefits": [
    {
      "name": "General Admission",
      "description": "Full-day access to the festival grounds.",
      "rules": "Valid for single entry. Present QR code at the gate.",
      "startDate": "2026-07-20T10:00:00.000Z",
      "qrCode": "https://app.w3block.io/redeem/benefit-a1b2c3d4"
    }
  ]
}
```

### Step 2: Send the Request

**Endpoint:**

| Method | Path | Auth |
|--------|------|------|
| POST | `/pass/generateTickets` | None |

**Example Request:**

```bash
curl -X POST "https://pdf.w3block.io/pass/generateTickets" \
  -H "Content-Type: application/json" \
  -d '{
    "tokenName": "Summer Festival Pass",
    "tokenEditionNumber": "1",
    "tokenVariants": [{"label": "Tier", "value": "VIP"}],
    "benefits": [{
      "name": "General Admission",
      "description": "Full-day access to the festival.",
      "rules": "Single entry. Present QR at gate.",
      "startDate": "2026-07-20T10:00:00.000Z",
      "qrCode": "https://app.w3block.io/redeem/abc123"
    }]
  }' \
  -o pass.pdf
```

### Step 3: Handle the Response

**Success (200):**

The response is a PDF binary stream with the following headers:

```
Content-Type: application/pdf
Content-Disposition: attachment; filename=pass.pdf
```

The file is always returned as an attachment (download), unlike certificate endpoints which support inline preview.

**Error Responses:**

| Status | Cause | Action |
|--------|-------|--------|
| 400 | Missing required fields (`tokenName`, `tokenEditionNumber`, `benefits`) or malformed DTO | Validate request body against the PassDto schema. Ensure `benefits` array is not empty |
| 500 | PDF generation failed (template error, Puppeteer crash) | Check service logs. Verify template syntax if using a custom template |
| 408 | Generation exceeded 30-second timeout | Reduce the number of benefits. Simplify custom templates. Ensure QR code data is not excessively long |

## Built-in Template Features

When no `pdfTemplate` is provided, the service uses its built-in template located at `/src/templates/template.html`. This template provides a complete pass layout with the following features:

### Layout Structure

1. **Header Section**
   - Displays `tokenName` prominently
   - Shows `tokenEditionNumber` below the name

2. **Variants Section**
   - Iterates through `tokenVariants` array
   - Each variant is displayed as a label/value pair (e.g., "Tier: VIP", "Color: Gold")
   - Skipped entirely if `tokenVariants` is an empty array

3. **Benefits Section**
   - Iterates through each benefit in the `benefits` array
   - Each benefit card displays:
     - **QR Code**: Generated client-side via QRCode.js from the `qrCode` data string
     - **Name**: Benefit title
     - **Description**: Full description text
     - **Rules**: Usage rules and restrictions
     - **Dates**: Start date and end date (when provided), formatted for readability
     - **Check-in Times**: Start and end check-in windows (when provided)

4. **Page Break Control**
   - The template uses CSS `page-break-inside: avoid` on benefit cards
   - This prevents a single benefit card from being split across two pages
   - Each benefit occupies a self-contained block

### QR Code Generation

The built-in template includes an embedded JavaScript section that generates QR codes at render time:

- Uses the QRCode.js library (bundled in the template)
- Each benefit's `qrCode` string is encoded into a QR code image
- QR codes are rendered as canvas elements and converted to inline images
- The QR code size is optimized for print scanning

## Flow: Generate Pass with Custom Template

Override the built-in template by providing a `pdfTemplate` field in the request body.

### Step 1: Create the Custom Template

Write a full HTML document using Handlebars syntax. The following variables are available:

| Variable | Type | Description |
|----------|------|-------------|
| `tokenName` | string | Token pass display name |
| `tokenEditionNumber` | string | Edition number or label |
| `tokenVariants` | array | Array of `{ label, value }` objects |
| `benefits` | array | Array of benefit objects (see BenefitsDto) |

**Example Custom Template:**

```html
<html>
<head>
  <style>
    body {
      font-family: 'Helvetica Neue', Arial, sans-serif;
      margin: 0;
      padding: 30px;
      color: #1a1a1a;
    }
    .header {
      text-align: center;
      border-bottom: 2px solid #c9a54e;
      padding-bottom: 20px;
      margin-bottom: 30px;
    }
    .header h1 {
      font-size: 32px;
      margin: 0;
    }
    .header .edition {
      font-size: 16px;
      color: #888;
      margin-top: 5px;
    }
    .variants {
      display: flex;
      justify-content: center;
      gap: 20px;
      margin-bottom: 30px;
    }
    .variant {
      background: #f5f5f5;
      padding: 8px 16px;
      border-radius: 4px;
    }
    .variant .label {
      font-weight: bold;
      color: #666;
    }
    .benefit {
      border: 1px solid #e0e0e0;
      border-radius: 8px;
      padding: 20px;
      margin-bottom: 20px;
      page-break-inside: avoid;
    }
    .benefit h2 {
      margin-top: 0;
      color: #c9a54e;
    }
    .benefit .rules {
      background: #fff8e1;
      padding: 10px;
      border-radius: 4px;
      font-size: 13px;
    }
    .qr-container {
      text-align: center;
      margin: 15px 0;
    }
  </style>
  <script src="https://cdn.jsdelivr.net/npm/qrcodejs@1.0.0/qrcode.min.js"></script>
</head>
<body>
  <div class="header">
    <h1>{{tokenName}}</h1>
    <div class="edition">Edition {{tokenEditionNumber}}</div>
  </div>

  {{#if tokenVariants.length}}
  <div class="variants">
    {{#each tokenVariants}}
    <div class="variant">
      <span class="label">{{this.label}}:</span> {{this.value}}
    </div>
    {{/each}}
  </div>
  {{/if}}

  {{#each benefits}}
  <div class="benefit">
    <h2>{{this.name}}</h2>
    <p>{{this.description}}</p>
    <div class="rules">{{this.rules}}</div>
    <p>
      <strong>From:</strong> {{dateFormat this.startDate "DD/MM/YYYY HH:mm"}}
      {{#if this.endDate}}
        <strong>To:</strong> {{dateFormat this.endDate "DD/MM/YYYY HH:mm"}}
      {{/if}}
    </p>
    {{#if this.startCheckinTime}}
    <p>
      <strong>Check-in:</strong>
      {{dateFormat this.startCheckinTime "HH:mm"}} - {{dateFormat this.endCheckinTime "HH:mm"}}
    </p>
    {{/if}}
    <div class="qr-container" id="qr-{{@index}}"></div>
  </div>
  {{/each}}

  <script>
    document.addEventListener('DOMContentLoaded', function() {
      var benefits = {{{json benefits}}};
      benefits.forEach(function(benefit, index) {
        new QRCode(document.getElementById('qr-' + index), {
          text: benefit.qrCode,
          width: 180,
          height: 180
        });
      });
    });
  </script>
</body>
</html>
```

## Integration with Pass Module

The pass PDF generation service is typically called by the Pass & Benefits backend, not directly by end users. The integration flow is:

```
User purchases       Pass backend         PDF service
token-pass      -->  creates benefits  -->  generates PDF
                     with QR codes          with QR codes
                                       <--  returns PDF stream
                <--  delivers PDF
                     to user
```

### How the Pass Backend Calls the PDF Service

1. The Pass backend collects the token pass data (name, edition, variants) and benefit data (including QR code redemption URLs)
2. It sends a `POST /pass/generateTickets` request to the PDF service
3. The PDF service generates the PDF and returns it as a binary stream
4. The Pass backend delivers the PDF to the user (via download link, email attachment, etc.)

### QR Code Contents

Each benefit's `qrCode` field typically contains a redemption URL that encodes:
- The benefit ID
- The token pass edition
- A verification token or signature

When scanned at a check-in point, the URL resolves to the W3Block platform's benefit redemption flow.

## Error Handling

### Common Error Scenarios

| Scenario | Error | Resolution |
|----------|-------|------------|
| Empty `benefits` array | 400 BadRequestException | At least one benefit must be provided in the request |
| Missing `tokenName` or `tokenEditionNumber` | 400 BadRequestException | Both fields are required strings |
| Benefit missing `qrCode` field | 400 BadRequestException | Every benefit must include a `qrCode` string |
| Custom template has invalid Handlebars syntax | 500 InternalServerErrorException | Validate template syntax. Check for unclosed blocks (`{{#if}}`, `{{#each}}`) |
| PDF generation exceeds 30-second timeout | 408 RequestTimeoutException | Reduce benefit count, simplify template, or compress embedded resources |
| QRCode.js fails to load in custom template | 500 InternalServerErrorException | Include the QRCode.js library via CDN or inline it in the template |

### Timeout Considerations

The 30-second global timeout applies to the entire PDF generation process, including:
- Template rendering with Handlebars
- QR code generation via JavaScript execution
- Puppeteer page rendering and PDF conversion

Passes with many benefits (10+) or complex custom templates with heavy CSS may approach this limit. Strategies to stay within the timeout:

1. Limit benefits to a reasonable number per PDF (recommended: under 15)
2. Use simple CSS layouts (avoid complex flexbox/grid with many nested elements)
3. Optimize QR code sizes (the default 180x180px is a good balance)
4. Avoid loading external resources (fonts, images) from slow CDNs

## Common Pitfalls

| # | Pitfall | Detail |
|---|---------|--------|
| 1 | QR code data string is too long | Extremely long URLs or data strings produce dense QR codes that may be difficult to scan. Keep QR data under 500 characters. Use short URLs or UUIDs instead of full data payloads |
| 2 | Many benefits produce a large PDF | Each benefit adds roughly one page. A pass with 20 benefits generates a 20+ page PDF, which takes longer to generate and may hit the timeout. Batch benefits into separate PDFs if needed |
| 3 | Custom template missing QR code script | If you use a custom `pdfTemplate`, you must include the QR code generation JavaScript yourself. The built-in QRCode.js is only included in the default template. Include it via CDN or inline the library |
| 4 | Page breaks splitting benefit cards | Use `page-break-inside: avoid` on benefit card containers to prevent a single benefit from being split across two pages. The built-in template already handles this |
| 5 | Date formatting in custom templates | Use the `dateFormat` Handlebars helper for date display (e.g., `{{dateFormat this.startDate "DD/MM/YYYY HH:mm"}}`). Raw ISO 8601 strings are not user-friendly |
| 6 | Empty `tokenVariants` array still required | Even if there are no variants, the field must be present as an empty array `[]`. Omitting it entirely causes a validation error |
